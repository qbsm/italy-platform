# CHANGELOG и семантическое версионирование релизов

## CHANGELOG

- Файл: **CHANGELOG.md** в корне проекта.
- Формат: [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/) (секции Added, Changed, Deprecated, Removed, Fixed, Security).
- Записи группировать по версиям; внутри версии — по типу изменения.
- Каждая версия привязана к дате в формате `YYYY-MM-DD`.

Пример:

```markdown
## [1.1.0] - 2025-02-23

### Added

- Health check endpoint GET /health.
- Rate limiting для POST /api/send.

### Changed

- ...

## [1.0.0] - 2025-01-15

...
```

Ссылки `[1.1.0]` при необходимости ведут на тег в репозитории (например `https://github.com/.../compare/1.0.0...1.1.0`).

## Семантическое версионирование (SemVer)

Формат: **MAJOR.MINOR.PATCH** (например `1.2.3`).

- **MAJOR** — несовместимые изменения API или поведения (ломающие изменения).
- **MINOR** — новая функциональность с обратной совместимостью.
- **PATCH** — исправления багов, обратно совместимые.

Для веб-платформы: изменение контракта API (api/send), удаление или переименование маршрутов, смена формата данных в JSON для страниц — учитывать как MAJOR или MINOR в зависимости от договорённостей с потребителями. Обычные правки контента в `data/json/` к версии не привязывают или учитывают как PATCH при релизе «обновление контента».

## Теги и релизы

- Тег в Git: `v1.1.0` (с префиксом `v`).
- Релиз: создать тег после обновления CHANGELOG и `version` в package.json (и при необходимости в иных местах, где хранится версия).

Команды:

```bash
# Обновить CHANGELOG.md и version, затем:
git tag -a v1.1.0 -m "Release 1.1.0"
git push origin v1.1.0
```
